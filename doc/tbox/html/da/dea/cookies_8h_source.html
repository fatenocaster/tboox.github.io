27 /* //////////////////////////////////////////////////////////////////////////////////////
28  * INTRODUCTION
29  * Cookies are a general mechanism which server side connections (such as CGI scripts)
30  * can use to both store and retrieve information on the client side of the connection.
31  * The addition of a simple, persistent, client-side state significantly extends the capabilities
32  * of Web-based client/server applications.
33  *
34  * OVERVIEW
35  * A server, when returning an HTTP object to a client, may also send a piece of state information
36  * which the client will store. Included in that state object is a description of the range of
37  * URLs for which that state is valid. Any future HTTP requests made by the client which fall
38  * in that range will include a transmittal of the current value of the state object from the client back to the server.
39  * The state object is called a cookie, for no compelling reason.
40  * This simple mechanism provides a powerful new tool which enables a host of new types of applications to
41  * be written for web-based environments. Shopping applications can now store information about the currently
42  * selected items, for fee services can send back registration information and free the client from
43  * retyping a user-id on next connection, sites can store per-user preferences on the client,
44  * and have the client supply those preferences every time that site is connected to.
45  *
46  * SPECIFICATION
47  * A cookie is introduced to the client by including a Set-Cookie header as part of an HTTP response,
48  * typically this will be generated by a CGI script.
49  * Syntax of the Set-Cookie HTTP Response Header
50  * This is the format a CGI script would use to add to the HTTP headers a new piece of data which is to be stored
51  * by the client for later retrieval.
52  * Set-Cookie: NAME=VALUE; expires=DATE;
53  * path=PATH; domain=DOMAIN_NAME; secure
54  *
55  * NAME=VALUE
56  * This string is a sequence of characters excluding semi-colon, comma and white space.
57  * If there is a need to place such data in the name or value, some encoding method such as URL style %XX encoding
58  * is recommended, though no encoding is defined or required.
59  * This is the only required attribute on the Set-Cookie header.
60  *
61  * expires=DATE
62  * The expires attribute specifies a date string that defines the valid life time of that cookie.
63  * Once the expiration date has been reached, the cookie will no longer be stored or given out.
64  * The date string is formatted as:
65  *
66  * Wdy, DD-Mon-YYYY HH:MM:SS GMT
67  * This is based on RFC 822, RFC 850, RFC 1036, and RFC 1123, with the variations that the only legal time zone
68  * is GMT and the separators between the elements of the date must be dashes.
69  * expires is an optional attribute. If not specified, the cookie will expire when the user's session ends.
70  *
71  * Note: There is a bug in Netscape Navigator version 1.1 and earlier. Only cookies whose path attribute
72  * is set explicitly to "/" will be properly saved between sessions if they have an expires attribute.
73  *
74  *
75  * domain=DOMAIN_NAME
76  * When searching the cookie list for valid cookies, a comparison of the domain attributes of the cookie
77  * is made with the Internet domain name of the host from which the URL will be fetched. If there is a tail match,
78  * then the cookie will go through path matching to see if it should be sent.
79  * "Tail matching" means that domain attribute is matched against the tail of the fully qualified domain name
80  * of the host. A domain attribute of "acme.com" would match host names "anvil.acme.com" as well as
81  * "shipping.crate.acme.com".
82  * Only hosts within the specified domain can set a cookie for a domain and domains must have at least two (2)
83  * or three (3) periods in them to prevent domains of the form: ".com", ".edu", and "va.us".
84  * Any domain that fails within one of the seven special top level domains listed below only require two periods.
85  * Any other domain requires at least three. The seven special top level domains are:
86  * "COM", "EDU", "NET", "ORG", "GOV", "MIL", and "INT".
87  *
88  * The default value of domain is the host name of the server which generated the cookie response.
89  *
90  *
91  * path=PATH
92  * The path attribute is used to specify the subset of URLs in a domain for which the cookie is valid.
93  * If a cookie has already passed domain matching, then the pathname component of the URL is compared with
94  * the path attribute, and if there is a match, the cookie is considered valid and is sent along with the URL request.
95  * The path "/foo" would match "/foobar" and "/foo/bar.html". The path "/" is the most general path.
96  * If the path is not specified, it as assumed to be the same path as the document being described
97  * by the header which contains the cookie.
98  *
99  *
100  * secure
101  * If a cookie is marked secure, it will only be transmitted if the communications channel with the host
102  * is a secure one. Currently this means that secure cookies will only be sent to HTTPS (HTTP over SSL) servers.
103  * If secure is not specified, a cookie is considered safe to be sent in the clear over unsecured channels.
104  *
105  * max-age
106  * A positive valueindicates that the cookie will expire after that many seconds have passed.
107  * Note that the value is the maximum age when the cookie will expire, not the cookie's current age.
108  * A negative value means that the cookie is not stored persistently and will be deleted when the Web browser exits.
109  * A zero value causes the cookie to be deleted.
110  * The default value is -1
111  *
112  * Syntax of the Cookie HTTP Request Header
113  * When requesting a URL from an HTTP server, the browser will match the URL against all cookies
114  * and if any of them match, a line containing the name/value pairs of all matching cookies will be included
115  * in the HTTP request. Here is the format of that line:
116  * Cookie: NAME1=OPAQUE_STRING1; NAME2=OPAQUE_STRING2 ...
117  *
118  * Additional Notes
119  * Multiple Set-Cookie headers can be issued in a single server response.
120  *
121  * Instances of the same path and name will overwrit each other, with the latest instance taking precedence.
122  * Instances of the same path but different names will add additional mappings.
123  *
124  * Setting the path to a higher-level value does not override other more specific path mappings.
125  * If there are multiple matches for a given cookie name, but with separate paths,
126  * all the matching cookies will be sent. (See examples below.)
127  *
128  * The expires header lets the client know when it is safe to purge the mapping but the client is not required to do so.
129  * A client may also delete a cookie before it's expiration date arrives if the number of cookies exceeds
130  * its internal limits.
131  *
132  * When sending cookies to a server, all cookies with a more specific path mapping should be sent before cookies
133  * with less specific path mappings. For example, a cookie "name1=foo" with a path mapping of "/" should be
134  * sent after a cookie "name1=foo2" with a path mapping of "/bar" if they are both to be sent.
135  *
136  * There are limitations on the number of cookies that a client can store at any one time.
137  * This is a specification of the minimum number of cookies that a client should be prepared to receive and store.
138  * 300 total cookies
139  * 4 kilobytes per cookie, where the name and the OPAQUE_STRING combine to form the 4 kilobyte limit.
140  * 20 cookies per server or domain. (note that completely specified hosts and domains are treated as
141  * separate entities and have a 20 cookie limitation for each, not combined)
142  * Servers should not expect clients to be able to exceed these limits. When the 300 cookie limit
143  * or the 20 cookie per server limit is exceeded, clients should delete the least recently used cookie.
144  * When a cookie larger than 4 kilobytes is encountered the cookie should be trimmed to fit,
145  * but the name should remain intact as long as it is less than 4 kilobytes.
146  *
147  * If a CGI script wishes to delete a cookie, it can do so by returning a cookie with the same name,
148  * and an expires time which is in the past. The path and name must match exactly in order for the expiring cookie
149  * to replace the valid cookie. This requirement makes it difficult for anyone but the originator of a cookie
150  * to delete a cookie.
151  *
152  * When caching HTTP, as a proxy server might do, the Set-cookie response header should never be cached.
153  *
154  * If a proxy server receives a response which contains a Set-cookie header,
155  * it should propagate the Set-cookie header to the client, regardless of whether the response was 304 (Not Modified)
156  * or 200 (OK).
157  * Similarly, if a client request contains a Cookie: header, it should be forwarded through a proxy,
158  * even if the conditional If-modified-since request is being made.
159  *
160  * EXAMPLES
161  * Here are some sample exchanges which are designed to illustrate the use of cookies.
162  * First Example transaction sequence:
163  * Client requests a document, and receives in the response:
164  * Set-Cookie: CUSTOMER=WILE_E_COYOTE; path=/; expires=Wednesday, 09-Nov-99 23:12:40 GMT
165  * When client requests a URL in path "/" on this server, it sends:
166  * Cookie: CUSTOMER=WILE_E_COYOTE
167  * Client requests a document, and receives in the response:
168  * Set-Cookie: PART_NUMBER=ROCKET_LAUNCHER_0001; path=/
169  * When client requests a URL in path "/" on this server, it sends:
170  * Cookie: CUSTOMER=WILE_E_COYOTE; PART_NUMBER=ROCKET_LAUNCHER_0001
171  * Client receives:
172  * Set-Cookie: SHIPPING=FEDEX; path=/foo
173  * When client requests a URL in path "/" on this server, it sends:
174  * Cookie: CUSTOMER=WILE_E_COYOTE; PART_NUMBER=ROCKET_LAUNCHER_0001
175  * When client requests a URL in path "/foo" on this server, it sends:
176  * Cookie: CUSTOMER=WILE_E_COYOTE; PART_NUMBER=ROCKET_LAUNCHER_0001; SHIPPING=FEDEX
177  * Second Example transaction sequence:
178  * Assume all mappings from above have been cleared.
179  *
180  * Client receives:
181  * Set-Cookie: PART_NUMBER=ROCKET_LAUNCHER_0001; path=/
182  * When client requests a URL in path "/" on this server, it sends:
183  * Cookie: PART_NUMBER=ROCKET_LAUNCHER_0001
184  * Client receives:
185  * Set-Cookie: PART_NUMBER=RIDING_ROCKET_0023; path=/ammo
186  * When client requests a URL in path "/ammo" on this server, it sends:
187  * Cookie: PART_NUMBER=RIDING_ROCKET_0023; PART_NUMBER=ROCKET_LAUNCHER_0001
188  * NOTE: There are two name/value pairs named "PART_NUMBER" due to the inheritance of the "/" mapping
189  * in addition to the "/ammo" mapping.
190  *
191  */
